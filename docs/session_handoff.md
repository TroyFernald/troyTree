# Troy Family Tree Research Project - Session Handoff

## Purpose

This repo is a local genealogy research and cleanup system for the Troy family
tree. The source tree came from an Ancestry GEDCOM export. The system should not
rewrite Ancestry or treat public web trees as proof. It imports GEDCOM data,
preserves original records, creates candidate evidence, generates review reports,
and supports human review before any fact is accepted.

Repository pushed to GitHub:

`https://github.com/TroyFernald/troyTree`

## Working Philosophy

- Ancestry remains the source and record-review platform.
- Google Drive/local exports store files.
- SQLite stores imported data, evidence assertions, review targets, and reports.
- Public web sources are clues, not proof.
- Repeated names across generations are normal; never merge on name alone.
- Royal/noble/medieval lines are high-risk until proven through the colonial bridge.

## Key Source Files

- `src/init_database.py`  
  Database schema and migrations.

- `src/gedcom_import.py`  
  Parses `Troy Tree.ged`, imports people/families, preserves raw records, creates
  citations and evidence assertions.

- `src/import_pilot_data.py`  
  Main pipeline entry. Runs GEDCOM import, pilot CSV import, duplicate detection,
  direct ancestor audit, validation, web research target/finding import, notable
  candidate generation, and review task generation.

- `src/export_reports.py`  
  Writes CSV/XLSX/Markdown reports to `data/exports/`.

- `src/direct_ancestor_audit.py`  
  Tags root/direct ancestors and creates focused research queue items.

- `src/duplicate_detection.py`  
  Candidate duplicate detection. Important: same name alone is not enough.
  Conflicting years and generation gaps are penalized/blocked.

- `src/relationship_validation.py`  
  Checks death before birth, parent too young/old, child after parent death,
  self-parent, ancestor cycles, and date parse issues.

- `src/web_research.py`  
  Generates whole-family public search targets and imports seeded web findings.

- `src/notable_people.py`  
  Detects notable-person candidates: Mayflower, colonial public figures,
  military titles, royal/noble/medieval claims.

- `config/web_research_seeds.yaml`  
  Seeded public findings for the first weakly sourced direct ancestors.

- `docs/genealogy_database_best_practices.md`  
  Research/modeling notes: evidence vs conclusion, citation modeling, review
  workflow, privacy/versioning.

## Known Caveats

- The GEDCOM import parser is lightweight and custom. It captures core INDI/FAM
  data and raw records but does not fully model every GEDCOM tag yet.
- `people` still stores imported working profile facts directly. The better long
  term model is to treat those as imported conclusions or working display values
  and use `evidence_assertion` + `conclusion_evidence` for proof.
- The web research workflow currently imports seeded findings and generates
  search targets; it does not yet automatically query/search public web APIs.
- Generated reports are tracked in git because the user explicitly approved
  including project data. Be careful before pushing private genealogy data to any
  other remote.
- Pytest may warn about `.pytest_cache` permission under this environment. Tests
  still pass.
